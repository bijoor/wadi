# Tauri desktop-app packaging for the Konkan-house editor

## Why now (and why before the Claude skill)

The editor + viewer today assume a developer setup: clone repo, `npm run
build`, `python3 -m http.server`, open `localhost:8000/editor/`. An
architect can't distribute or install that. Everything downstream —
Claude-skill authoring, sharing configs with clients, presenting to
builders — is gated on being able to hand someone an installer.

Tauri is the shortest path from "React+Three.js SPA that works" to
"double-click .dmg / .msi / .AppImage that runs offline":

- The editor is already a pure client-side Vite bundle; Tauri consumes
  Vite output verbatim.
- Native file dialogs replace copy/paste of JSON — architects work in
  Save-As / Open-Recent instead.
- Native webview means no CORS friction, no browser cache staleness,
  no service-worker gotchas.
- Rust binary is small (~5-15 MB) vs Electron's 100+ MB.
- Auto-update is built in.

## Non-goals

- **Server-side rendering / SSR**: everything stays client-side. Nothing
  to migrate.
- **Rewriting the SVG/roof pipeline in Rust**: TypeScript stays. Rust
  is only for OS integration (file I/O, dialogs, updater, menus).
- **Mobile builds**: Tauri supports iOS/Android but out of scope for
  v1. Focus on macOS + Windows first, Linux third.

## What to change in the editor code

Minimal but not zero. Key sites:

1. **File I/O** — `editor/src/io/fileIO.ts` currently uses a `<input type="file">`
   pick + `Blob` download. Add a Tauri-aware branch:
   ```ts
   const isTauri = typeof window !== "undefined" && "__TAURI__" in window;
   if (isTauri) {
     const path = await open({ filters: [{ name: "House", extensions: ["json"] }] });
     const text = await readTextFile(path);
     ...
   } else {
     // existing browser flow
   }
   ```
   Same pattern for `downloadConfig`. Keeps the browser build working
   (docs deployment stays functional as a fallback / preview URL).

2. **Templates** — currently fetched via `fetch("templates/index.json")`
   from the built `docs/editor/templates/` folder. In Tauri, templates
   are bundled as **Rust resources** and exposed via a
   `#[tauri::command] fn list_templates()` returning the JSON. Add
   the same "is Tauri?" branch in the load flow.

3. **Base path** — Vite's default `base: './'` works in Tauri (asset
   URLs are relative). Confirm during first build.

4. **Native menu bar** — File → New / Open / Save / Save As / Recent,
   Edit → Undo / Redo, View → Toggle 3D / 2D / Roof. Wire menu items
   to existing store actions (`useConfigStore` methods) via Tauri
   event listeners.

5. **Window title reflects file state** — "Wadi — untitled *"
   / "Wadi — my_home.json". Store's `filename` + `dirty`
   already track this; just plumb into `appWindow.setTitle(...)`.

That's ~200 LOC of TypeScript changes + a small `src-tauri/src/main.rs`.

## Build + distribution

### Development
```bash
cd editor && npm install
cd ../src-tauri && cargo install tauri-cli
cd .. && cargo tauri dev       # hot-reload frontend + Rust
```

### Production build
```bash
cargo tauri build              # produces installers under src-tauri/target/release/bundle/
```

Outputs per platform:
- **macOS**: `.dmg` (universal binary; needs Apple Developer ID for
  distribution outside dev machines — see Signing).
- **Windows**: `.msi` + `.exe`. Needs code-signing cert to avoid
  SmartScreen warnings; unsigned works but shows scary dialog on
  first launch.
- **Linux**: `.AppImage` + `.deb`. No signing needed.

### Signing & notarization

- **macOS**: Apple Developer Program ($99/yr), Developer ID
  Application certificate, notarize via `xcrun altool` (Tauri has an
  action for GH Actions). Without it, users get "app is damaged"
  scare and must right-click → Open.
- **Windows**: EV code-signing cert (~$200-400/yr from DigiCert /
  Sectigo). Without it, SmartScreen flags the installer for weeks
  until enough downloads build reputation.
- **Linux**: no signing needed; `.AppImage` runs directly.

Recommendation for v1: ship unsigned + document the right-click-Open
workaround. Add signing before the app hits >~10 non-technical users.

### Auto-update

Tauri has a built-in updater — points at a JSON manifest
(`latest.json`) served from a static URL (GitHub Releases works).
Bundled installers include public key; new releases are signed with
the matching private key. Set up once, then `cargo tauri build`
signs each release automatically. See tauri.app/v2/plugin/updater.

## Rollout phases

### Phase T1 — Local desktop-app shell (1-2 days)
- `cargo tauri init` in the repo.
- Point Tauri at existing `editor/dist/`.
- `cargo tauri dev` runs the editor in a native window.
- No file dialogs yet — user still uses browser file picker via
  webview.
- **Ship target**: internal dogfood — you double-click, see the
  editor in a real window.

### Phase T2 — Native file I/O (3-5 days)
- Add Tauri commands for `open_config`, `save_config`, `save_as`,
  `list_recent`.
- Fork `io/fileIO.ts` with the Tauri branch.
- Native menu bar with File → Open/Save/Recent/New.
- Persist recent-files list in Tauri's `app_data_dir`.
- **Ship target**: architects can open a `.json` from Finder /
  Explorer and save changes without paste-in.

### Phase T3 — Bundled templates + New-House flow (2-3 days)
- Templates embedded as Rust resources.
- `File → New from template →` submenu using existing template list.
- Cross-platform icon assets (`cargo tauri icon`).
- **Ship target**: user installs, launches, picks a template, edits,
  saves. No web-server, no template folder to manage.

### Phase T4 — Export flows (2-3 days)
- File → Export → SVG (Master Roof Sheet / Floor Plan / Elevation)
  writes files via native save dialog.
- Existing SVG generators already produce strings — just add file-write
  glue in the Tauri command layer.
- **Ship target**: end-of-workflow output goes straight to disk instead
  of `svgMap.get(...)` in DevTools.

### Phase T5 — Distribution (3-5 days incl. iteration)
- GitHub Actions workflow: on tag, build for mac + win + linux, upload
  to Releases.
- Draft install docs (right-click Open workaround, checksum verification).
- **Ship target**: `.dmg` / `.msi` download link + one-page install
  guide. First architect installs from a link, no CLI.

### Phase T6 — Signing + auto-update (only when needed)
- Apple Developer + Windows cert.
- Wire Tauri updater manifest.
- **Ship target**: subsequent updates land automatically.

Total: ~2-3 weeks of intermittent work for T1-T5, T6 later.

## Where this puts the Claude Skill

Once T2 ships, the Skill workflow becomes:

1. Architect drops sketch into Claude Chat with the skill installed.
2. Claude produces JSON.
3. Architect clicks the copy button → opens the desktop app → File
   → New → pastes.
4. Iterate: architect screenshots the 3D view back to Claude.

Once we add an MCP server (Phase 3 of the Claude Skill plan), it can
run INSIDE the Tauri app — expose the same commands over stdio /
websocket, and Claude iterates directly against the running app.
That's the endgame: Claude sees the current config, proposes a
patch, the app validates + renders, all in one loop.

## Open questions

- **Bundle size budget**: Tauri + WebView2 (Windows) + notarized
  binary — target < 20 MB installed. Vite bundle is currently ~2 MB
  gzipped so should be fine.
- **File-format ownership**: register `.house.json` as a file type
  Tauri owns? Nice UX but requires OS-integration work per platform.
  Defer to T7.
- **Multiple documents open at once?** Tauri supports multi-window;
  our Zustand store is single-config. Simpler to keep single-window
  for v1 (matches how most CAD apps work anyway).
- **Cloud sync / collaboration?** Not for v1. Users share `.json`
  files by any medium. If needed later, add a Cowork-style backend.

## Risks

- **WebView2 dependency on Windows**: pre-installed on Windows 11 +
  most Windows 10 with recent updates. Older Win10 machines need the
  Evergreen bootstrapper. Tauri bundles it; installer is ~10 MB
  bigger.
- **macOS notarization pipeline**: first-time setup takes half a day
  of Apple portal wrangling. Docs are fine but rough.
- **The Rust learning curve**: v1 needs ~50 LOC of Rust across
  `main.rs` + 4-5 commands. Not deep, but a new tool if you haven't
  touched Rust.
- **GitHub Actions minutes**: cross-platform builds are ~15 min each
  on free tier. Public repo → free.
